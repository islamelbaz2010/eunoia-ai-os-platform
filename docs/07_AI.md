# 07 — AI

## Overview

The AI system is a RAG (Retrieval-Augmented Generation) pipeline. It allows users to ask natural language questions and receive answers grounded in their organization's knowledge base documents. No answers are invented — the system either cites sources or says it doesn't know.

  if (!client) {
    client = new OpenAI({
      apiKey: env.OPENAI_API_KEY,

## Chunking Strategy

**File**: `src/lib/ai/chunk.ts`

```
Constants:
  CHUNK_SIZE    = 1000 characters
  CHUNK_OVERLAP = 150 characters

Algorithm:
  1. Normalize CRLF → LF, trim
  2. Sliding window: start=0
     - end = min(start + 1000, text.length)
     - push text[start..end].trim()
     - if end == text.length → break
     - start = end - 150  ← overlap ensures context continuity
  3. Filter empty chunks
```

**Example** for 1150-char text:
- Chunk 1: chars 0–1000 (1000 chars)
- Chunk 2: chars 850–1150 (300 chars, with 150-char overlap from chunk 1)

**Why 1000/150?**
- 1000 chars ≈ 200–250 words — enough semantic context per chunk
- 150-char overlap prevents splitting mid-sentence from losing context
- text-embedding-3-small is token-based (not char-based), but ~1000 chars ≈ ~250 tokens, well within the model's 8191-token limit

**Why not semantic chunking?**
- Simpler, deterministic, no dependencies
- Adequate for hospitality FAQs and SOPs (typically well-structured paragraphs)
- Can upgrade to semantic chunking in Phase 2

## Ingestion Pipeline

**File**: `src/lib/ai/ingest.ts`

```
ingestDocument({ documentId, organizationId, content })
  │
  ├── 1. chunkText(content) → string[]
  │         Sliding window, 1000 chars, 150 overlap
  │
  ├── 2. DELETE FROM knowledge_base_chunks WHERE document_id = documentId
  │         Re-ingestion: remove old chunks first (idempotent)
  │
  ├── 3. if chunks.length === 0 → return early
  │
  ├── 4. embedTexts(chunks) → number[][]
  │         Batched in groups of 512
  │         POST to OpenAI /embeddings
  │         Returns 1536-dim float array per chunk
  │
  └── 5. INSERT INTO knowledge_base_chunks
              { document_id, organization_id, content, embedding }
              Throws on error (caller handles with document cleanup)
```

**Failure handling in `createDocument()` (knowledge-base/actions.ts)**:
```
INSERT knowledge_base_documents → get id
TRY ingestDocument(id, ...)
CATCH:
  DELETE knowledge_base_documents WHERE id = data.id  ← cleanup orphan
  return { error: "Failed to index document..." }
```

This ensures no unindexed document row persists after an embedding failure.

## Vector Storage

- **Table**: `knowledge_base_chunks`
- **Column**: `embedding vector(1536) NOT NULL`
- **Index**: `knowledge_base_chunks_embedding_idx` — **HNSW** with `vector_cosine_ops`

HNSW (Hierarchical Navigable Small World) provides approximate nearest-neighbor search. It is faster than exact search (IVFFlat) for high-dimensional vectors and has excellent recall at k=6.

## Retrieval Pipeline

**File**: `src/app/dashboard/assistant/actions.ts`

```
askAssistant(question: string)
  │
  ├── 1. verifySession() + getActiveOrganization()
  │
  ├── 2. Validate question: 3–500 chars (Zod)
  │
  ├── 3. embedText(question) → 1536-dim vector
  │         Uses same model as ingestion: text-embedding-3-small
  │
  ├── 4. supabase.rpc("match_kb_chunks", {
  │         query_embedding: JSON.stringify(queryEmbedding),
  │         target_org_id: org.id,
  │         match_count: 6
  │       }) → [{ id, document_id, content, similarity }]
  │
  ├── 5. Filter: similarity >= 0.3  (MIN_SIMILARITY)
  │         Chunks below this threshold are likely irrelevant.
  │         Returns early if 0 chunks pass the threshold.
  │
  ├── 6. Assemble context:
  │         "[1] chunk content\n\n[2] chunk content\n\n..."
  │
  ├── 7. GPT-4o-mini completion:
  │         system: "You are Eunoia, an AI assistant for a hospitality property.
  │                  Answer only using the provided context.
  │                  Cite sources with [1], [2], etc.
  │                  If the context doesn't contain the answer, say you don't know
  │                  — never invent information."
  │         user: "Context:\n{context}\n\nQuestion: {question}"
  │         max_tokens: 1024
  │
  └── 8. logUsageEvent("rag_query") fire-and-forget
```

## Embedding API Call Details

**Single text embedding** (`embedText`):
```typescript
openai.embeddings.create({
  model: "text-embedding-3-small",
  input: text,      // string
})
// Returns: response.data[0].embedding  (number[1536])
```

**Batch embedding** (`embedTexts`):
```typescript
// Splits into batches of 512
openai.embeddings.create({
  model: "text-embedding-3-small",
  input: batch,     // string[]
})
// Returns: response.data.map(d => d.embedding)  (number[][1536])
```

## Similarity Threshold Rationale

`MIN_SIMILARITY = 0.3` (cosine similarity where 1.0 = identical, 0.0 = orthogonal, -1.0 = opposite)

- Cosine similarity with text-embedding-3-small: typical values
  - High relevance: 0.7–0.9
  - Moderate relevance: 0.4–0.7
  - Low relevance / noise: 0.1–0.3
  - Unrelated: < 0.2
- 0.3 is intentionally conservative — the goal is to return no answer rather than a hallucinated answer based on vaguely related content
- Can be tuned based on user feedback

## Token Budget

| Operation | Typical cost |
|-----------|-------------|
| Embedding one chunk (1000 chars ≈ 250 tokens) | 250 tokens |
| Embedding a document (50 chunks) | 12,500 tokens ≈ $0.00025 |
| RAG query embedding | ~100 tokens |
| RAG context (6 chunks × 250 tokens) | 1,500 tokens |
| RAG completion (GPT-4o-mini, system+context+answer) | ~2,000 input + 500 output ≈ $0.0006 |

**Cost per document indexed**: ~$0.0003
**Cost per RAG query**: ~$0.001
**Monthly cost at 1000 queries/day**: ~$30

## System Prompt Design

The system prompt is intentionally constrained:
1. **Identity**: "You are Eunoia, an AI assistant for a hospitality property."
2. **Grounding**: "Answer only using the provided context."
3. **Citations**: "Cite sources with [1], [2], etc."
4. **Refusal**: "If the context doesn't contain the answer, say you don't know — never invent information."

This is a strict grounding prompt. It prevents hallucinations by explicitly forbidding the model from drawing on training data outside the context window.

## Known Limitations

1. **No streaming**: The `chat.completions.create()` call is non-streaming. The user sees "Thinking..." until the full response is ready. For long answers this can be 3–10 seconds.

2. **No conversation history**: Each question is independent — there is no multi-turn context. The context window only contains retrieved chunks + current question.

3. **Sources not rendered**: `askAssistant()` returns `{ answer, sources }` but `chat.tsx` only displays `answer`. The sources array (with similarity scores) is discarded.

4. **Text-only ingestion**: KB documents must be pasted as text. PDF, DOCX, HTML, and URL-based ingestion are not supported.

5. **No language-aware retrieval**: The `language` field on documents is stored but not used in retrieval. A question in Arabic will embed into Arabic vector space and may not retrieve Arabic documents optimally if they were embedded separately.

6. **No reranking**: Pure HNSW search with no cross-encoder reranking. For a hospitality use case this is usually adequate but may degrade for long, complex queries.

7. **No chunk metadata**: Chunks don't store paragraph position, heading context, or section metadata. This limits citation quality.
