## Protocol

The new route returns `text/event-stream` with three event types:

```
data: {"type":"sources","sources":[{...}]}   ← arrives before first token
data: {"type":"delta","content":"Hello"}      ← one per token
data: {"type":"delta","content":" world"}
data: {"type":"done"}                         ← stream complete
data: {"type":"error","message":"..."}        ← on server-side failure
```

Sources are sent before the first token — the user sees what the AI is drawing
from while the answer is still being written.

## UX Transformation

**Before:**
- User clicks Ask
- Button greys out
- "Thinking..." appears
- 5-6 second freeze with zero feedback
- All text appears at once

**After:**
- User clicks Ask
- Placeholder message appears immediately
- Sources appear within ~1 second (embed + search)
- Text streams token by token (blinking cursor visible)
- Button label changes to "Generating…" during stream
- Sources collapsible panel ready when answer completes

## Business Impact

This is a demo-critical fix. A potential Customer #1 evaluating the product now sees:

1. An instant response (placeholder appears in milliseconds)
2. Progressive disclosure — sources appear first, then the answer builds
3. A blinking cursor indicating active generation
4. A premium AI feel — indistinguishable from ChatGPT or Claude in terms of responsiveness

Without streaming, a 5-second freeze during a demo triggers hesitation.
With streaming, the product feels alive, fast, and trustworthy.

## Security

- Rate limiting: identical to the original (50 queries/user/hour before any OpenAI call)
- Auth: `verifySession()` + `getActiveOrganization()` called at start of every request
- Input validation: same Zod schema as the original
- Org isolation: `organization_id` always from session membership, never from client
- `server-only` import at top of route file
- Usage logging: fire-and-forget in `finally` block — never blocks the response

---

## Technical Notes

- `X-Accel-Buffering: no` header disables nginx/proxy buffering (required for SSE on Vercel)
- `Cache-Control: no-cache, no-transform` prevents any proxy from caching the stream
- AbortController in the client cleans up on unmount
- Partial error handling: if an error event arrives after some content was already streamed,
  the partial message is kept (content was already shown to user)
